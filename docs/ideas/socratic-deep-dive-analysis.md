# 코딩 테스트 소크라테스 딥다이브 (Socratic Deep-Dive) - 기획 분석 및 가설 검증

본 분석은 `pm-product-discovery` 스킬의 `identify-assumptions-new` 프레임워크를 기반으로 작성되었습니다. 기획된 'AI 면접관 딥다이브' 프로덕트가 실패할 수 있는 잠재적 위험을 3가지 관점(PM, 디자인, 엔지니어링)에서 다각도로 분석하고, 8가지 리스크 카테고리에 맞춰 핵심 가설과 테스트 방법을 도출했습니다.

---

## 1. 다각도 관점의 리스크 평가 (3 Perspectives)

- **제품 관리자 (Product Manager)**
  - **관심사:** 유저들이 정말로 코딩 테스트 직후의 면접 딥다이브를 원할까? '해커톤 심사위원'을 타겟으로 한 "Wow Factor"로는 훌륭하지만, 실제 제품으로 출시될 경우 면접 준비생들이 유료(혹은 지속적)로 사용할 만큼의 Pain Point를 확실히 해결해 주는가?
  - **위험 요소:** 핵심 알고리즘 패턴 5개에 집중하는 전략은 해커톤용으론 완벽하지만 장기적 리텐션 확보에는 부족할 수 있음.
- **디자이너 (Designer)**
  - **관심사:** 인지적 부하(Cognitive Load). 코드를 짜는 것 자체만으로도 에너지가 많이 소모되는데, AI 면접관이 집요하게 4-5 depth로 물고 늘어질 때 유저가 느끼는 피로감.
  - **위험 요소:** 답변을 강제하거나 지나치게 어려우면 사용자는 이탈(Drop-off)함. 끝까지 대화를 완주하고 '액셔너블 피드백 리포트'를 받아보는 비율이 현저히 낮을 수 있음.
- **엔지니어 (Engineer)**
  - **관심사:** LangGraph 상태 제어의 복잡도와 LLM 할루시네이션(Hallucination). 제출된 코드의 퀄리티가 예상 외로 굉장히 엉망이거나, 창의적인(그러나 성능은 떨어지는) 로직일 때 AI가 이를 제대로 이해하고 유의미한 꼬리 질문을 던질 수 있는가?
  - **위험 요소:** 프롬프트만으로 정적 코드 분석을 대체하기엔 AI의 답변 일관성이 떨어질 수 있어, 백엔드 의존성(AST 기반 힌트 등)이 필수적으로 요구될 가능성이 있음.

## 2. 8대 위험 카테고리별 핵심 가설 및 검증 방법 (8 Risk Categories)

### 🥇 1. Value (가치)
- **가설 (Assumption):** 사용자는 다른 면접 준비 툴(단순 모의 면접)보다 '자신의 코드 기반에 맞춘 심층 꼬리 질문'을 통해 실제 면접 능력이 크게 향상되었다고 느낄 것이다.
- **신뢰도 (Confidence):** 보통 (Medium)
- **검증 테스트 (Test):** 타겟 사용자 5명에게 본인의 과거 코드에 대해 사람이 직접 딥다이브 면접을 진행한 후 문제 해결력 향상도와 만족도를 평가한다(Wizard of Oz 테스트).

### 🎨 2. Usability (사용성)
- **가설 (Assumption):** 코딩으로 에너지를 소진한 유저가 '압박 질문'에 지쳐 중도 이탈하지 않고, 3~4턴의 대화를 끝까지 마칠 것이다. (문서 내 핵심 가설)
- **신뢰도 (Confidence):** 낮음 (Low)
- **검증 테스트 (Test):** 데모 챗봇 UI를 구축하여 문제 제출 직후 채팅 면접 창을 띄우고, 각 턴(Turn)마다 유저의 중도 이탈률(Drop-off rate)을 측정한다. 게임 요소(Gamification)나 시각적 진행바(Progress bar) 도입 고민 필요.

### 💰 3. Viability (사업성/실행가능성)
- **가설 (Assumption):** 과도한 Judge 서버나 수백 개의 DB 구축을 포기하고, 오직 '에이전트 기술력(LangGraph)'만으로 심사위원들을 설득할 수 있다.
- **신뢰도 (Confidence):** 높음 (High)
- **검증 테스트 (Test):** 해커톤/서바이벌 심사 기준표에서 '기술적 난이도 및 AI 활용도' 비중을 재확인하고 사전 모의 피칭을 통해 "왜 다른 기능을 포기하고 여기에 집중했는가"에 대한 논리를 검증한다.

### ⚙️ 4. Feasibility (기술적 구현 가능성)
- **가설 (Assumption):** LLM은 지원자 코드의 비정상 로직을 정확히 짚어내고 사실(Fact) 기반의 오류 없는 꼬리 질문을 4-5 depth까지 생성할 수 있다.
- **신뢰도 (Confidence):** 낮음~보통 (Low-Medium)
- **검증 테스트 (Test):** 성능이 떨어지는 코드 20개를 준비해 직접 Claude/GPT 프롬프트 체인에 넣고 유의미한 꼬리 질문이 생성되는지 백엔드/AI 엔지니어가 블라인드 평가한다. (AST 등 정적 분석 데이터 필요 여부 조기 결정)

### ⚖️ 5. Ethics (윤리성)
- **가설 (Assumption):** AI는 지원자의 특이한(그러나 정답인) 코드 스타일이나 마이너한 패러다임을 부당하게 감점하거나 비난하지 않고 공정하게 대화할 것이다.
- **신뢰도 (Confidence):** 높음 (High)
- **검증 테스트 (Test):** 동일 구조이나 다른 스타일(절차지향 기반 vs 함수형 기반)의 정답 코드를 입력했을 때, 프롬프트의 반응 온도가 공정한지 테스트한다.

### 🚀 6. Go-to-Market (GTM & 출시)
- **가설 (Assumption):** '특화 알고리즘 5개'라는 적은 풀(Pool)로도 지원자들의 이목을 끌 바이럴 콘텐츠(또는 포트폴리오 데모)를 만들 수 있다.
- **신뢰도 (Confidence):** 보통 (Medium)
- **검증 테스트 (Test):** 대표적인 알고리즘(예: 단기간에 성취감을 느끼기 쉬운 Two Pointer 문제) 1개로 데모 비디오를 제작해 랜딩 페이지에 올리고, 사전 신청자 클릭률(CTR)을 확인한다.

### 🎯 7. Strategy & Objectives (전략 및 목표)
- **가설 (Assumption):** 넓고 얕은 기능(다양한 문제 세트, 완벽한 채점기 등)보다, 좁지만 깊은 기능( LangGraph 기반 다중 턴 면접관)이 "AI Champion" 서바이벌 우승에 더 적합한 포지셔닝이다.
- **신뢰도 (Confidence):** 높음 (High)
- **검증 테스트 (Test):** 과거 AI 경진 대회 우승 사례 혹은 멘토링 피드백을 통해 심사위원이 선호하는 '혁신의 깊이(Depth of Innovation)' 수준을 체크한다.

### 🤝 8. Team (팀 역량)
- **가설 (Assumption):** 팀 내에 LangGraph 상태 머신 설계와 효율적인 프롬프트 체이닝, 최소 기능 Sandbox를 해커톤 기간 내에 병렬로 개발할 수 있는 역량 배분이 되어 있다.
- **신뢰도 (Confidence):** 보통 (Medium)
- **검증 테스트 (Test):** 첫 개발 1~2일 내에 LangGraph의 단일 턴 로직(루팅→질문 생성)만을 테스트하는 PoC(Proof of Concept)를 최우선적으로 완료하여 기술적 병목을 미리 찾아낸다.

## 3. 핵심 우선순위 제안 (Next Steps)
1. **오픈 퀘스천(# Open Questions) 해소 촉구:** 현재 기획서상 제일 큰 불확실성은 **"프롬프트만 의존 vs 정적 분석 데이터(AST 등) 주입"**입니다. 이는 기술적 구현 가능성(Feasibility) 및 기간 내 완성의 성패를 가르므로 즉시 기술 검증(PoC)에 착수하는 것을 권장합니다.
2. **유저 이탈 방지 장치(UX) 고안:** 4-5번의 압박 질문을 버텨내게 할 넛지(Nudge), 예를 들어 답변 시 힌트 기능 도입 혹은 단계별로 채워지는 시각적 성취 이펙트 등 '사용성(Usability)' 강화 장치가 구체적으로 구상되어야 합니다.
