# 코딩 테스트 - 기술 면접 소크라테스 딥다이브 (Socratic Deep-Dive)

## Problem Statement 
> "우리가 어떻게 하면 코딩 테스트 통과자들이 자신이 작성한 코드의 논리와 트레이드오프를 면접관에게 자신 있게 설명할 수 있도록 기술 면접(Technical Interview) 대비의 간극을 자동으로 메워줄 수 있을까?"

## Recommended Direction (핵심 방향성)
**오답 및 최적화 유도형 딥다이브 AI 면접관 (Socratic Deep-Dive)**
코딩 테스트 문제를 푼 뒤 결과(통과/실패)에 관계없이 지원자 코드를 기반으로 깊이 있는 기술 면접을 진행합니다. AI는 절대 정답을 먼저 말해주지 않으며, 지원자가 스스로 빈틈이나 최적화 방안을 깨닫도록 유도합니다. 
예를 들어 *"N이 10만을 넘어갈 때 이 코드는 어떻게 작동할까요?", "현재 작성하신 O(N^2) 코드를 O(N)으로 줄일 수 있는 다른 자료구조가 생각나시나요?"*라며 4-5 depth까지 집요하게 물고 늘어집니다. 
이를 통해 AI R&D 서바이벌(AI Champion) 심사에서 **'방향성이 있는 다중 턴 추론 및 상태(State) 제어 전략'**이라는 기술적 고도화를 강력하게 어필합니다.

## Key Assumptions to Validate (반드시 검증해야 할 핵심 가설)
- [ ] **기술적 추론력:** LLM(Claude/GPT)이 제출된 지원자 코드의 기상천외한 로직이나 엣지 케이스 오류를 정확하게 파악하고 유의미한 꼬리 질문을 동적으로 생성할 수 있다. (대회에서 가장 공격받을 환각/할루시네이션 리스크)
- [ ] **유저 잔존율(UX):** 코딩으로 에너지를 쓴 유저들이 AI의 '압박 질문'에 지쳐 이탈하지 않고 끝까지 대화 세션을 마칠 것이다.
- [ ] **기술적 차별성:** 심사위원들은 단순 프롬프팅이 아닌 본 솔루션의 에이전트 구조(LangGraph 기반 상태 머신 등)를 기술적으로 진일보했다고 평가할 것이다.

## MVP Scope (AI 챔피언 제출용 핵심 범위)
- 코딩 테스트 경험용 웹 기반 기초 에디터 및 코드 제출 폼
- 핵심을 찌르기 좋은 특정 알고리즘 패턴(ex: BFS/DFS, DP, Two Pointer) 특화 문제 5~10개 엄선 제공
- 답안 제출 → LangGraph 기반 "질문 에이전트" 활성화 → 3~4턴의 소크라테스식 채팅 면접 파이프라인
- 딥다이브 종료 후 제공받는 [논리 구조 / 기술 정확도 / 소통 능력] 3축 기반의 고퀄리티 액셔너블 피드백 리포트 (외부 공유 링크 포함)

## Not Doing (and Why) (하지 않을 것과 그 이유)
- **코딩 작성 중 실시간 개입 (Pair-programming)**
  - *이유:* 코딩의 맥을 끊어 UX 호불호가 갈리며, 에디터 타이핑 상태를 실시간 스트리밍(WebSockets)으로 동기화하는 개발 공수가 해커톤 기한 대비 너무 높습니다. 우선순위에서 배제합니다.
- **네카라쿠배 등 수백 개의 기업별 맞춤 기출 DB 구축**
  - *이유:* 방대한 문제 수집은 데이터베이스 싸움이지 AI R&D 대회의 핵심인 'AI 기술력 측정'이 아닙니다. 확실하게 딥다이브가 동작하는 패턴 5개만 확보하여 AI의 'Wow Factor'를 보여주는 데 집중합니다.
- **과도하게 완벽한 채점 서버(Judge) 자체 개발**
  - *이유:* 핵심 경쟁력은 '알고리즘 채점'이 아니라 '코드 설명 면접'입니다. 채점은 최소한의 보안만 갖춘 가벼운 Python Sandbox로 구현하거나 외부 API, 심지어 모킹(Mocking)으로 대체해 리소스를 절약해야 합니다.

## Open Questions (개발 전 결정해야 할 논의점)
- **LLM 구조 설계:** AI가 코드의 취약점을 파고들 때, 프롬프트(LLM)에 모든 코드 분석을 의존할 것인가? 아니면 백엔드에서 AST(추상 구문 트리) 등을 활용하여 **정적 분석 데이터를 힌트로 RAG처럼 LLM에 주입**하여 정확도를 극한으로 끌어올릴 것인가?
