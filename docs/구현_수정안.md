# Socratic Deep-Dive 구현 수정안

> 목적: 현재 기획과 구현 제안서가 가진 약점을 정리하고, AI Champion 대회에서 더 설득력 있는 우승권 서사로 재정렬하기 위한 수정 기준 문서
> 기준 문서: `docs/proposal_template.md`, `docs/platform_plan_v2.md`, `docs/구현_제안서.md`

## 문서 사용법

- 이 문서는 새 아이디어 브레인스토밍 문서가 아니라, 기존 `구현_제안서.md`를 더 강한 방향으로 고치기 위한 수정 기준이다.
- 핵심 원칙은 바꾸지 않는다.
  - coding test to interview bridge
  - hallucination control
  - three-axis evaluation
- 바꾸는 것은 "누구의 어떤 문제를 어떤 구조로 해결하는가"에 대한 서사 우선순위다.
- 결론부터 말하면, 이 프로젝트는 `취준생용 AI 면접 연습 서비스`보다 `교육기관/훈련기관용 통제형 기술면접 평가 시스템`으로 포지셔닝하는 편이 훨씬 강하다.

## 1. 일반 사항 수정안

### 1.1 현재 문서의 약점

- 현재 문서는 문제의식 자체는 타당하지만, 심사위원 입장에서 "좋은 기능 조합"처럼 읽힐 위험이 있다.
- `코테를 풀고 AI 면접을 본다`는 흐름은 이해되지만, 왜 이 프로젝트가 꼭 필요한지에 대한 시장 압력이 약하다.
- `취준생`, `이직 준비 개발자`, `부트캠프`, `대학`, `B2B2C`가 동시에 등장해 핵심 고객이 흐려진다.
- 결과적으로 "재미있는 AI 데모"는 되지만 "산업적 필요성이 높은 AI 시스템"으로는 약해진다.

### 1.2 수정 원칙

- 제안서의 1차 고객을 `교육기관`, `부트캠프`, `대학 취업지원 조직`, `기업 내부 개발자 교육 프로그램`으로 고정한다.
- 최종 사용자로서의 학습자는 유지하되, 구매자와 운영 주체는 기관으로 잡는다.
- 문제 정의를 개인의 불안이 아니라 기관의 운영 비용과 평가 품질 문제로 번역한다.

### 1.3 수정된 문제 정의

현재보다 더 강한 문제 정의는 아래와 같다.

> 교육기관과 훈련기관은 수강생의 코딩 테스트 결과를 볼 수는 있지만, 실제 기술 면접에서 중요한 "설명 능력", "복잡도 이해", "트레이드오프 방어 능력"을 대규모로 반복 평가하기 어렵다.  
> 사람 면접관이 일일이 수행하기에는 시간과 비용이 많이 들고, 범용 LLM을 그대로 사용할 경우 질문 일관성과 사실 정확도를 통제하기 어렵다.  
> 따라서 본 기술은 `코딩 테스트 결과 직후의 기술 설명 역량`을, `통제 가능한 질문 구조` 안에서 반복 가능하게 평가하는 시스템을 목표로 한다.

### 1.4 기술명/활용분야 수정 방향

기술명은 현재 명칭을 유지해도 되지만, 부제가 필요하다.

권장 표현:

- `Socratic Deep-Dive`
- 부제: `코딩 테스트 기반 통제형 AI 기술면접 평가 시스템`

활용분야 권장 표현:

- IT/SW 교육
- 채용 준비
- 개발자 역량 평가 자동화
- 교육기관용 AI 평가 인프라

### 1.5 유사기술 / 차별점 수정 방향

현재 유사기술 비교는 나쁘지 않지만, 차별점이 아직 기능 나열형이다.  
차별점은 반드시 아래 3축으로 정리하는 편이 좋다.

1. `통제형 질문 생성`
   - 문제별 Fact/Trap/Boundary를 기준으로 질문 주제 이탈을 제한
   - 범용 챗봇과 달리 질문의 재현 가능성과 검수 가능성을 확보

2. `코드 기반 평가 연결`
   - 사용자의 제출 코드 AST 신호를 질문 분기와 연결
   - 단순 면접 질문이 아니라 "왜 이 질문이 나왔는지"를 코드 근거로 설명 가능

3. `기관 운영 가능성`
   - 저비용 텍스트 기반 인터뷰 구조
   - 수강생별 약점 데이터 축적 가능
   - 향후 강사 override, 문항 authoring, cohort 분석으로 확장 가능

### 1.6 일반 사항에 바로 반영할 문장 예시

`1.7 차별점`에는 아래 톤을 권장한다.

> 본 기술은 범용 대화형 AI 면접 도구와 달리, 문제별 Fact/Trap/Follow-up Goal 데이터를 기준으로 질문 범위를 통제하고, 제출 코드의 경량 AST 분석 결과를 질문 흐름에 직접 반영하는 `통제형 기술면접 평가 시스템`이다.  
> 이를 통해 기관은 대규모 학습자에게 일관된 면접 훈련 경험을 제공하면서도, 질문 정확도와 운영 비용을 동시에 관리할 수 있다.

## 2. 기술 명세 수정안

### 2.1 현재 문서의 약점

- 현재 구조 설명은 기술적으로 그럴듯하지만, 핵심 엔진이 무엇인지 한 번에 안 들어온다.
- `AI 면접관`이 주인공처럼 보이는데, 실제 핵심은 `면접관이 아니라 질문 통제 구조`다.
- 데모 시연 포인트와 시스템 아키텍처 포인트가 아직 분리되어 있지 않다.

### 2.2 기술목적 수정 방향

`기술목적`은 취준생의 감정 문제보다 기관의 평가 문제를 중심으로 써야 한다.

단, 기존 비전 문장("모든 개발자가 자신의 코드를 자신 있게 설명할 수 있는 세상")은 영감을 주는 북극성으로서의 가치가 있으므로 완전히 삭제하지 않는다.  
권장하는 방식은 **비전(Why)과 전략(How)의 2단 구조**다:

- **비전 (Why):** "모든 개발자가 자신의 코드를 자신 있게 설명할 수 있는 세상을 만든다."
- **전략 (How):** "이를 위해 교육기관이 대규모로 도입 가능한 통제형 기술면접 평가 시스템을 구축한다."

비전까지 B2B 톤으로 바꾸면 "기관 납품 사업"처럼 보일 수 있고, 대회의 혁신성 평가 축에서 손해를 볼 수 있다.  
비전은 감성적으로, 전략은 기관 중심으로 — 둘 다 한 문단 안에 넣는 것이 최적이다.

권장 구조:

1. 기존 코딩 테스트는 정답 여부 위주 평가에 머문다.
2. 실제 기술 면접의 핵심인 설명 능력 평가는 대규모 자동화가 어렵다.
3. 범용 LLM은 질문 통제, 사실 정확도, 비용 구조 측면에서 기관 도입이 어렵다.
4. 본 기술은 코드 제출 직후의 설명 역량을 통제 가능한 방식으로 평가하는 시스템이다.

권장 문장 예시:

> 본 기술의 목적은 코딩 테스트 이후 단계에서 요구되는 기술 설명 역량을 반복 가능하고 통제 가능한 방식으로 평가하는 것이다.  
> 기존 코딩 테스트 플랫폼은 정답 여부와 실행 결과를 중심으로 평가하며, 교육기관은 수강생의 설명 능력을 확인하기 위해 여전히 사람 면접관의 시간을 대량 투입해야 한다.  
> 본 기술은 문제별 Fact/Trap 데이터와 코드 기반 분석을 결합하여, 기관이 저비용으로 기술면접 훈련과 약점 진단을 운영할 수 있도록 한다.

### 2.3 기술구조 수정 방향

현재 구조에서 가장 강조해야 할 것은 아래 흐름이다.

1. `문제 가이드 레이어`
   - optimal solution
   - traps
   - follow-up goals
   - forbidden boundaries

2. `코드 프로파일 레이어`
   - parse success/failure
   - nested loop depth
   - risky ops
   - recursion pattern
   - cyclomatic complexity

3. `인터뷰 오케스트레이션 레이어`
   - Normal Flow
   - Plan B Flow
   - Fallback Flow

4. `평가 및 리포트 레이어`
   - logical structure
   - technical accuracy
   - explanation clarity

즉, 시스템 구조도는 "LLM이 답변한다"가 아니라 아래처럼 보여야 한다.

```text
문제 선택/코드 제출
→ 코드 프로파일 추출(AST)
→ 문제별 Fact/Trap 가이드 로딩
→ 상태 기반 질문 오케스트레이션
→ 다중 턴 면접 세션
→ 3축 리포트 생성
→ 기관/학습자용 피드백 자산화
```

### 2.4 주요기능 수정 방향

주요기능은 사용자 기능이 아니라 시스템 가치 중심으로 정리하는 편이 좋다.

권장 묶음:

1. `통제형 질문 생성 기능`
   - 문제 범위 이탈 방지
   - Trap 기반 follow-up questioning

2. `코드 근거 기반 분기 기능`
   - AST 신호에 따라 Normal / Plan B / Fallback 전환

3. `3축 평가 기능`
   - 논리 구조
   - 기술 정확도
   - 설명 명확도

4. `기관 운영 확장 기능`
   - 세션 저장
   - 약점 유형 축적
   - 향후 강사 검수/대시보드 확장 가능

### 2.5 혁신적 요소 수정 방향

혁신 포인트는 "AI를 썼다"가 아니라 "AI를 어떻게 제어했는가"다.

반드시 강조해야 할 문장:

- 본 기술의 혁신은 자유 대화형 AI 면접이 아니라 `검수 가능한 질문 생성 구조`에 있다.
- AI가 무엇을 묻지 말아야 하는지까지 구조적으로 제한한다.
- 코드 프로파일과 문제 가이드를 결합해 `질문 생성의 근거`를 남긴다.

### 2.6 도전적 요소 수정 방향

도전성은 다음처럼 써야 강하다.

- 높은 품질의 자연스러운 질문을 만들면서도, 질문 범위를 통제해야 한다.
- 사용자 코드 품질이 극단적으로 좋거나 나쁜 경우에도 세션이 무너지지 않아야 한다.
- 외부 API 의존성이 흔들려도 fallback 시나리오로 데모 흐름을 유지해야 한다.

즉, 도전 과제는 모델 성능 자체보다 `제어`, `안정성`, `운영 가능성`이다.

## 3. 구현 방법 및 계획 수정안

### 3.1 현재 문서의 약점

- 현재 구현 범위는 MVP 수준으로 타당하지만, "무엇을 증명할 것인가"보다 "무엇을 만들 것인가"에 치우쳐 있다.
- 대회 심사에서는 기능 수보다 검증 설계가 더 중요하다.
- 지금 문서는 개발 계획처럼 보이지만, 우승권 서사에는 `실험 설계`와 `데모 비교 구조`가 더 들어가야 한다.

### 3.2 구현범위 수정 원칙

기능 개발 항목보다 아래 4개 증명 항목 중심으로 재정렬한다.

1. `질문 통제 가능성 증명`
   - 목표 지표: AI 질문의 주제 이탈률 ≤ 5% (Fact/Trap 범위 밖 질문 비율)
   - 측정: 5개 패턴 × 3회 반복 시행으로 질문 범위 준수 여부 확인

2. `코드 기반 분기 타당성 증명`
   - 목표 지표: 동일 문제에서 코드 품질이 다를 때 분기(Normal/PlanB/Fallback) 정확 전환율 ≥ 90%
   - 측정: 좋은 코드 / 나쁜 코드 / 파싱 실패 코드 3종 세트 비교

3. `리포트 일관성 증명`
   - 목표 지표: 동일 입력에 대한 리포트 3축 점수 편차 ≤ 15%
   - 측정: 같은 세션 로그를 3회 반복 평가하여 일관성 확인

4. `기관 운영 가능성 증명`
   - 목표 지표: 세션당 API 비용 ≤ $0.10, 면접 세션 완주율 ≥ 80%
   - 측정: 10~15세트 블라인드 테스트에서 비용·완주율 동시 검증

### 3.3 구현범위 권장안

#### 세부업무 #1: 문제별 Fact/Trap 평가 데이터 구축

- 핵심 5개 패턴 문제에 대해 Fact/Trap/Follow-up Goal/Forbidden Boundary 데이터를 구축
- 질문 품질보다 먼저 "질문 통제 가능성"을 검증
- 사람이 검토 가능한 JSON 규격으로 유지

#### 세부업무 #2: 코드 프로파일러 및 분기 엔진 구현

- Python AST 기반 코드 신호 추출
- nested loop, risky ops, recursion, complexity 등 핵심 시그널 정리
- 이 신호를 기반으로 Normal / Plan B / Fallback 분기 구현

#### 세부업무 #3: 멀티턴 면접 엔진 및 리포트 생성 구현

- 3~4턴 길이의 텍스트 기반 면접 세션 구현
- 질문 근거가 problem trap과 코드 프로파일 안에서 유지되는지 확인
- 세션 종료 후 3축 리포트 생성

#### 세부업무 #4: 대회용 검증/데모 패키지 구성

- 범용 LLM 방식과 제어형 방식의 차이를 보여주는 비교 시나리오 작성
- 데모 스크립트, 실패 대비 fallback 시나리오, Judge0 미연동 시 데모 모드 정리
- 기관 도입 관점의 운영 가치 설명 자료 구성

### 3.4 구현계획 수정 방향

현재 주차 계획은 유지 가능하지만, 각 주차의 산출물을 아래처럼 바꾸는 것이 좋다.

#### 1주차

- Fact/Trap JSON 3~5개 구축
- 단일 문제 기준 질문 생성 PoC
- "주제 이탈 없음" 확인

#### 2주차

- 코드 프로파일러와 분기 엔진 통합
- 좋은 코드 / 나쁜 코드 / 파싱 실패 코드 3종 세트 검증

#### 3주차

- 텍스트 인터뷰 UI와 리포트 UI 완성
- 데모 스토리 연결

#### 4주차

- 내부 블라인드 테스트
- 질문 정확도, 완주율, 리포트 일관성, 운영 스토리 정리
- 제출용 영상/문서 패키지 확정

### 3.5 기술스택 수정 방향

기술스택은 그대로 써도 되지만, 과장되게 보이지 않도록 아래 원칙을 따른다.

- 현재 구현된 것과 계획된 것을 분리해 쓴다.
- 미구현 기능은 roadmap 또는 향후 확장으로 명시한다.
- LangGraph는 실제로 필요해질 시점과 현재 custom state engine을 구분해 표현한다.

권장 방식:

- 현재: Next.js, FastAPI, Monaco Editor, Python AST, Judge0, Pydantic 기반 상태 관리
- 향후 확장: LangGraph, 기관 대시보드, 강사 override, cohort analytics

### 3.6 데모 수정 방향

대회용 데모는 반드시 아래 순서로 보여주는 편이 좋다.

1. 동일 문제에 대해 서로 다른 코드 제출
2. 코드 프로파일 차이 제시
3. 질문 플로우 차이 제시
4. 리포트 차이 제시
5. 기관이 얻는 데이터 자산 설명

즉, "한 사람이 AI랑 대화하는 장면"보다 `같은 문제인데 코드가 다르면 질문이 다르게 나온다`가 핵심 데모다.

### 3.7 즉시 수정해야 할 구현/문서 포인트

- 홈/데모 UI에서 `AI 면접 앱`처럼 보이는 요소보다 `코드 근거 기반 분기`와 `질문 통제`를 먼저 보이게 해야 한다.
- 랜딩 카피에서 감성적 표현보다 `기관이 왜 필요로 하는지`가 더 드러나야 한다.
- 데모 스크립트는 "재밌는 대화"가 아니라 "왜 이 질문이 나왔는지 설명 가능한가"를 중심으로 재작성해야 한다.

### 3.8 Fact/Trap 데이터 품질 보장 전략

현 팀에 알고리즘 도메인 전문가가 없다는 점은 인지하되, 이를 약점이 아니라 플랫폼 설계의 특성으로 프레이밍한다.

#### MVP 단계 (팀 직접 구축)

- 핵심 5개 패턴(Two Pointer, BFS, DP, Greedy, 정렬)에 대해 팀원이 **LLM 보조 + 교차 검수** 방식으로 Fact/Trap JSON 초안을 구축
- LeetCode, 프로그래머스 등 공개 해설 자료에서 주요 Trap 패턴을 추출하여 검수 근거로 활용
- 5개면 데모에 충분하다. 심사위원은 "5개도 이 품질이면 50개도 되겠구나"를 판단한다

#### Post-MVP (강사 위임 구조)

- 데이터 품질 보장의 주체를 **팀에서 기관(강사)으로 자연스럽게 이전**하는 설계
- 강사 저작 도구를 통해 기관별 전문가가 직접 Fact/Trap 데이터를 입력
- AI가 Trap 후보를 자동 생성 → 강사가 검수하는 Human-in-the-Loop 루프

제안서 권장 표현:

> MVP 단계에서는 팀이 직접 핵심 5개 패턴의 Fact/Trap을 구축·검수하고, 이후 강사 저작 도구를 통해 기관별 전문가가 직접 데이터를 입력하는 구조로 확장한다. 즉, 데이터 품질 보장의 주체가 팀에서 기관(강사)으로 자연스럽게 이전되는 플랫폼 설계다.

### 3.9 B2B 고객 검증 전략 (4주 내 실행 가능한 범위)

대회 단계에서는 실제 기관 계약이 아니라, **"기관 도입 시나리오의 설득력"**을 보여주면 된다.

#### 최소 검증 항목

1. **비용 비교 표 1장**
   - "강사 1명이 수강생 30명 기술 면접 = 약 60시간" vs "본 시스템 = 자동 반복 가능"
   - 부트캠프 커리큘럼 공개 자료에서 "모의 면접" 시간 비중을 인용하면 근거가 됨

2. **비공식 현장 인터뷰 1~2건** (선택, 가능하면 진행)
   - 주변 부트캠프 수강생/강사에게 카톡 5분 인터뷰
   - "모의 기술 면접 운영에 얼마나 시간이 드나요?"라는 질문 하나면 충분
   - 현장 목소리 한 줄이 제안서에 있으면 설득력이 급상승

3. **기관 도입 시나리오 서사**
   - 부트캠프 12주 커리큘럼 중 "모의 면접" 주차에 본 시스템을 투입하는 시나리오
   - 수강생 30명 × 5문제 = 150 세션을 강사 개입 없이 운영하는 흐름 제시

제안서 권장 표현:

> 국내 주요 부트캠프의 커리큘럼에는 '모의 기술 면접'이 필수 과정으로 포함되어 있으나, 이를 운영하는 데 강사 1인당 수강생 대비 상당한 시간이 소요된다. 본 시스템은 이 반복 비용을 구조적으로 줄이는 것을 목표로 한다.

## 4. 파급효과 수정안

### 4.1 현재 문서의 약점

- 파급효과가 맞는 방향이긴 하지만 아직 다소 넓고 추상적이다.
- "AI 에듀테크의 새 패러다임" 같은 말은 가능하지만, 심사위원이 보기엔 흔한 표현이 될 수 있다.

### 4.2 기술적 파급효과 수정 방향

기술적 파급효과는 아래처럼 좁고 구체적으로 쓰는 것이 좋다.

1. `통제형 질의응답 시스템` 설계 패턴 제시
2. `코드 프로파일 기반 평가 흐름 제어` 방식 제시
3. `LLM 환각 방어형 평가 구조`의 도메인 확장 가능성 제시

권장 문장 예시:

> 본 기술은 자유 대화형 LLM 평가가 가진 불안정성을 줄이기 위해, 문제별 제약 정보와 코드 기반 신호를 결합하여 질문 범위를 통제하는 평가 구조를 제시한다.  
> 이는 향후 개발자 교육뿐 아니라, 규칙 기반 상담 훈련, 전문직 시뮬레이션, 기업 내부 역량 평가 등 사실 정확도와 운영 일관성이 중요한 분야로 확장 가능한 구조다.

### 4.3 사회·산업적 파급효과 수정 방향

사회·산업적 파급효과는 개인 효용보다 기관 효용이 우선이다.

반드시 들어가야 할 포인트:

- 사람 면접관 시간 절감
- 반복 평가 가능
- 수강생 약점 데이터 축적
- 기관별 커리큘럼 개선 가능성

권장 문장 예시:

> 교육기관은 본 기술을 통해 수강생의 정답률뿐 아니라 설명 능력과 사고 과정까지 구조적으로 관찰할 수 있으며, 반복 면접 운영에 필요한 인력 부담을 줄일 수 있다.  
> 또한 세션 데이터가 축적되면 특정 cohort의 약점 패턴을 파악하여 커리큘럼 보완에 활용할 수 있으므로, 단발성 면접 도구가 아니라 교육 운영 인프라로 발전할 가능성이 있다.
