## 2. 기술 명세

| 2.1 기술목적 | **비전:** "모든 개발자가 자신의 코드를 자신 있게 설명할 수 있는 세상을 만든다."<br>**전략:** "교육기관이 대규모로 도입 가능한 통제형 기술면접 평가 시스템을 구축한다."<br><br>기존 코딩 테스트 플랫폼은 정답 여부와 실행 결과를 중심으로 평가하며, 교육기관은 수강생의 설명 능력을 확인하기 위해 여전히 사람 면접관의 시간을 대량 투입해야 한다. 범용 LLM을 그대로 활용할 경우 질문 통제, 사실 정확도, 비용 구조 측면에서 기관 도입이 어렵다.<br>본 기술은 코딩 테스트 이후 단계에서 요구되는 기술 설명 역량을 반복 가능하고 통제 가능한 방식으로 평가하는 것을 목적으로 한다. 문제별 Fact/Trap 데이터와 코드 기반 분석을 결합하여, 기관이 저비용으로 기술면접 훈련과 약점 진단을 운영할 수 있도록 한다.<br>- **B2B(교육 기관) 가치:** 부트캠프·대학 등에서 반복적으로 발생하는 기술 모의 면접 운영 부담을 줄이고, 수강생별 약점 데이터를 축적할 수 있는 저비용 텍스트 기반 면접 시스템<br>- **B2C(개발자) 가치:** 자신이 직접 작성한 코드 기반의 맞춤 압박 면접 훈련을 통해 설명 능력, 코드 최적화 센스, 트레이드오프 분석 역량을 종합적으로 향상 |
| --- | --- |
| 2.2 기술구조 | **[4계층 아키텍처]**<br>**1. 문제 가이드 레이어:** 문제별 optimal solution, follow-up goals, forbidden boundaries, traps를 구조화된 JSON으로 관리. 이 데이터가 LLM의 질문 생성 범위를 통제하는 '조종간(Fact)' 역할을 수행<br>**2. 코드 프로파일 레이어:** 제출된 코드의 경량 정적 분석(Python ast.NodeVisitor 기반: ①중첩 루프 깊이 감지 → 시간복잡도 질문 트리거, ②위험 연산 탐지(list.pop(0), sorted() 등) → Trap 매칭, ③재귀 호출 패턴 인식 → 기저 조건 질문 유도, ④Cyclomatic Complexity 산출). 코드 채점은 외부 SaaS(Judge0 API)를 활용하여 자체 채점 서버 구축 비용을 줄임<br>**3. 인터뷰 오케스트레이션 레이어:** 'Deterministic First' 원칙 기반 상태 분기 — CC ≥15 또는 파싱 실패 시 FallbackFlow(핵심 개념 질문), 위험 연산 0건 AND 중첩 루프가 낮은 경우 PlanBFlow(스케일업 압박), 그 외 NormalFlow(취약점 기반 꼬리질문). 면접관(Interviewer)은 주입된 문제 팩트와 코드 프로필 범위 안에서만 질문을 생성하며, 모델 미연동 시에는 규칙 기반 fallback 질문으로 세션을 지속<br>**4. 평가 및 리포트 레이어:** 3~4턴의 텍스트 문답 종료 후, 전체 대화 로그를 취합하여 논리 구조·기술 정확도·설명 명확도 기준의 3축 피드백 리포트를 생성. 모델 미연동 시에는 fallback 리포트로 대체. 세션 데이터는 기관/학습자용 피드백 자산으로 축적<br><br>**[구조도 요약]**<br>문제 선택/코드 제출 → 코드 프로파일 추출(AST) → 문제별 Fact/Trap 가이드 로딩 → 상태 기반 질문 오케스트레이션 (NormalFlow｜PlanBFlow｜FallbackFlow, 3~4턴) → 3축 종합 평가 → 기관/학습자용 피드백 자산화 |
| 2.3 주요기능 | **[MVP 핵심 기능 — 시스템 가치 중심]**<br>**1) 통제형 질문 생성 기능:** 문제별 Fact/Trap/Forbidden Boundary 데이터 범위 안에서만 질문을 생성하여 주제 이탈을 방지하고, Trap 기반 follow-up questioning으로 취약점을 정밀 탐색<br>**2) 코드 근거 기반 분기 기능:** AST 코드 프로파일 신호에 따라 NormalFlow / PlanBFlow / FallbackFlow 3가지 경로를 자동 전환. "왜 이 질문이 나왔는지"를 코드 근거로 설명 가능<br>**3) 3축 평가 기능:** 면접 종료 후 논리 구조 / 기술 정확도 / 설명 명확도 기반 3축 평가 리포트 자동 생성<br>**4) 기관 운영 확장 기능:** 세션 데이터 저장, 수강생별 약점 유형 축적, 향후 강사 검수/대시보드 확장 가능<br><br>**[Post-MVP 확장 기능]**<br>- *(신뢰도 확보)* B2B 도입 강사가 AI의 최종 평가를 보정할 수 있는 휴먼 오버라이드(Human-Override) 대시보드<br>- *(확장성 확보)* 강사 전용 커스텀 문항 Fact/Trap JSON 저작 도구 (AI Trap 후보 자동 생성 → 강사 검수)<br>- *(성장 장치)* '논리적 사고력 성장 그래프' 중심 SNS 공유용 피드백 리포트 카드 → B2B2C 바이럴 성장 엔진 |
| 2.4 결과물 형상 | 웹 기반 SaaS 플랫폼 (채용 기관, 교육 기관 타겟 B2B 애플리케이션 형태) |
| 2.5 배포방식 | 반응형 웹 애플리케이션 기반 배포 (프론트/백엔드/데이터 스토어 연동 클라우드 인프라 활용) |
| 2.6 혁신적 요소 | 본 기술의 혁신은 자유 대화형 AI 면접이 아니라 **'검수 가능한 질문 생성 구조'**에 있다. AI가 무엇을 묻지 말아야 하는지까지 구조적으로 제한하며, 코드 프로파일과 문제 가이드를 결합해 질문 생성의 근거를 남긴다.<br>- 프롬프트 자유도를 제약하고 문제별 Fact/Trap 데이터를 컨텍스트로 주입함으로써, 범용 대화형 LLM보다 문제 맥락에 맞는 면접 질문을 안정적으로 생성. AST 분석 결과와 규칙 기반 분기 로직을 결합하여 질문 흐름을 제어하는 구조가 핵심 차별점<br>- **Text-first MVP 기반 안정성 설계:** Monaco 기반 코드 제출, Judge0 연동 또는 Demo Mode, 모델 미연동 시 fallback 질문/리포트로 이어지는 구조를 통해 외부 의존성이 불안정해도 핵심 데모 흐름을 유지하도록 설계 |
| 2.7 도전적 요소 | 본 기술의 핵심 도전 과제는 모델 성능 자체보다 **제어, 안정성, 운영 가능성**이다.<br>**1) 질문 품질과 범위 통제의 양립:** 높은 품질의 자연스러운 질문을 만들면서도, Fact/Trap 데이터가 정의한 범위 밖으로 이탈하지 않도록 통제해야 한다.<br>**2) 양극단 엣지 케이스 처리:** 비정상적 스파게티 코드 입력 시 발동하는 '동적 폴백(Fallback) 핵심 개념 질문'과, 약점이 거의 없는 풀이 입력 시 발동하는 '시스템 스케일업 압박 질문(Plan B)' — 양극단을 하나의 상태 기반 분기 로직 안에서 자연스럽게 처리해야 한다.<br>**3) 외부 의존성 하의 데모 안정성:** 외부 API(Judge0·LLM) 미연동 상황에서도 Demo Mode와 fallback 질문/리포트로 세션을 계속 진행하여 시연 중 시스템 안정성을 확보해야 한다. 사전 준비된 결정론적 데모 스크립트(시나리오별 입출력 예시 완비)로 외부 API 불안정 상황에서도 핵심 가치를 빠짐없이 전달하며 시연을 완주할 수 있도록 설계되어 있다. |

## 3. 구현 방법 및 계획

| 3.1 구현범위 | **[구현 원칙: 기능 개발보다 4개 증명 항목 우선]**<br>**세부업무 #1: 문제별 Fact/Trap 평가 데이터 구축 (질문 통제 가능성 증명)**<br>핵심 5개 패턴 문제에 대해 Fact/Trap/Follow-up Goal/Forbidden Boundary 데이터를 구축. | 4.1 기술적 파급효과 | **1) 통제형 질의응답 시스템 설계 패턴 제시:** 본 기술은 자유 대화형 LLM 평가가 가진 불안정성을 줄이기 위해, 문제별 제약 정보와 코드 기반 신호를 결합하여 질문 범위를 통제하는 평가 구조를 제시합니다.<br>**2) 코드 프로파일 기반 평가 흐름 제어 방식 제시:** 기존 코딩 교육이 '정적 테스트 중심 패러다임'에 머물던 것에서, 코딩의 의도와 설명 능력을 묻는 '과정 지향적 Socratic 평가 패러다임'으로 AI 에듀테크 기술을 진일보시킵니다.<br>**3) LLM 환각 방어형 평가 구조의 도메인 확장 가능성 제시:** 이 구조는 향후 개발자 교육뿐 아니라, 규칙 기반 상담 훈련, 전문직 시뮬레이션, 기업 내부 역량 평가 등 사실 정확도와 운영 일관성이 중요한 분야로 확장 가능합니다.<br>**4) 자체 폐쇄형 Data Flywheel 기반 경쟁 진입장벽(Moat) 형성:** 기관 강사가 직접 저작·검수한 Trap 데이터는 해당 기관의 고유 평가 자산으로 Lock-in 효과를 창출합니다. 도입 기관이 늘수록 데이터 품질이 향상되고, 이는 다시 평가 신뢰도를 높이는 강화 피드백 루프를 형성합니다. 이 구조는 단순 SaaS 복제로는 재현할 수 없는 기술적·데이터적 진입장벽을 구성하며, 플랫폼이 성장할수록 후발 경쟁자의 추격 비용이 구조적으로 증가합니다. |
