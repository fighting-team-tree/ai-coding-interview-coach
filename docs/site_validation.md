# 현재 사이트 검증 메모

## 목적

- 현재 웹사이트가 `coding test to interview bridge`를 설득력 있게 보여주는지 검증한다.
- 특히 `교육기관/부트캠프` 관점에서 이 사이트가 실제로 도입 검토 가능한 수준의 제품처럼 보이는지 판단한다.
- 결과는 `지금 강한 점`, `신뢰를 깎는 지점`, `즉시 보완 우선순위`로 정리한다.

## 검증 기준

이번 검증은 아래 5개 축으로 고정한다.

| 축 | 질문 | 점수 |
| --- | --- | --- |
| 문제 인식 명확성 | 첫 화면만 보고 무엇을 해결하는지 이해되는가 | 3/5 |
| 코테→면접 브릿지 전달력 | 코드 제출 직후 설명 면접으로 이어지는 흐름이 자연스러운가 | 4/5 |
| 기관 구매자 가치 | 강사/부트캠프 운영자가 왜 필요한지 바로 보이는가 | 2/5 |
| 운영 신뢰성 | 질문 생성, 평가 방식, fallback 처리까지 믿을 만해 보이는가 | 2/5 |
| 데모 완결성 | 대표 시나리오가 중간 붕괴 없이 끝까지 완주되는가 | 4/5 |

## 한 줄 판정

현재 사이트는 `핵심 데모는 성립하지만 외부용 사이트로는 아직 부족하다`.

- 강한 부분은 대표 문제 상세 화면이다.
  - `코드 제출 → 질문 분기 → 질문 근거 → 3축 피드백`이 한 화면에서 이어진다.
  - 같은 문제에 다른 코드를 넣었을 때 질문 흐름이 달라진다는 데모 포인트가 명확하다.
- 약한 부분은 홈/랜딩이다.
  - 문서가 밀고 있는 `기관용 통제형 기술면접 평가 시스템`보다 `개인용 AI interview prep 서비스`처럼 읽힌다.
  - 따라서 부트캠프 강사나 기관 담당자는 “재미있는 데모”로는 보더라도 “도입할 시스템”으로는 바로 연결하기 어렵다.

## 근거 요약

### 1. 강한 지점

#### 문제 상세 화면의 제품 핵심은 잘 보인다

- `apps/web/components/problem-workspace.tsx`
  - `FLOW_META`로 `일반 심화 질문 / 확장 검증 질문 / 개념 복구 질문`을 명시한다.
  - `질문 근거`, `코드 분석 요약`, `왜 이런 질문이 나왔는지` 섹션이 있어 질문 제어 구조가 드러난다.
  - 대표 문제에서는 variant 버튼으로 `느린 풀이 / 안정적인 풀이 / 오류 코드`를 비교하게 되어 있어 데모 메시지가 선명하다.
- 이 구조는 문서에서 강조한 차별점과 정합적이다.
  - `docs/구현_제안서.md`: `통제형 질문 생성`, `코드 근거 기반 분기`, `3축 평가`
  - `docs/구현_수정안.md`: `AI 면접 앱처럼 보이는 요소보다 코드 근거 기반 분기와 질문 통제를 먼저 보이게 해야 한다`

#### 대표 사용자 흐름은 실제로 끝까지 돈다

- 실행 근거: `npm run qa:e2e`
- 확인된 사실
  - 홈에서 대표 CTA가 노출된다.
  - 대표 문제에서 코드 제출 후 `일반 심화 질문`이 시작된다.
  - 4턴 답변 뒤 `3축 피드백`까지 완료된다.
- 해석
  - 현재 구현은 “보여주려는 이야기”를 최소 한 번 끝까지 증명할 수 있는 수준이다.
  - 즉, 기능 부재보다 포지셔닝과 신뢰 신호 설계가 더 큰 문제다.

### 2. 약한 지점

#### 홈이 기관용 시스템이 아니라 개인 연습 서비스처럼 보인다

- `apps/web/app/page.tsx`
  - 브랜드가 `AI Champion Interview Prep`으로 노출된다.
  - 핵심 카피가 `문제를 푼 뒤 설명 면접까지 바로 이어서 연습합니다`로 시작한다.
  - 사이드 카드도 `최근 피드백 요약`, `오늘의 연습 흐름` 등 개인 사용 맥락이 강하다.
- 문제는 방향성이 틀렸다는 것이 아니라 우선순위가 틀렸다는 점이다.
  - 현재 문서의 1차 고객은 `교육기관/부트캠프`다.
  - 그런데 홈 첫인상은 `개인 학습 도구`로 읽힌다.

#### 기관 구매자 가치가 랜딩에서 거의 보이지 않는다

- 현재 홈에는 아래 질문에 대한 답이 바로 없다.
  - 왜 강사가 사람 면접관 시간을 줄일 수 있는가
  - 왜 질문 정확도를 믿어도 되는가
  - 왜 cohort 단위 운영 자산이 되는가
- 문서에는 이미 이 메시지가 정리돼 있다.
  - `docs/구현_제안서.md`: 기관 운영 가능성, 약점 데이터 축적, 운영 비용 관리
  - `docs/구현_수정안.md`: 기관 운영 pain을 먼저 보여야 한다
- 즉, 제품 정의는 있는데 랜딩 번역이 안 된 상태다.

#### 현재 기본 데모 경험은 fallback 의존성이 크고, 그 사실이 신뢰를 깎는다

- `apps/api/app/services/llm_service.py`
  - 모델 호출 실패 시 질문은 deterministic fallback으로, 리포트는 heuristic fallback으로 내려간다.
  - fallback summary가 영어로 시작한다: `Fallback report generated from transcript heuristics...`
- `apps/web/test/e2e/home-to-report.spec.ts`
  - E2E 테스트도 이 영문 fallback 문구를 기대값으로 사용한다.
- 이번 QA 실행 로그에서도 실제로 `OPENAI_API_KEY` 미설정으로 fallback이 반복되었다.
- 해석
  - 내부 QA 관점에서는 안전장치가 맞다.
  - 하지만 외부 시연이나 기관 검토 기준에서는 “현재 기본 체험이 임시 모드처럼 보인다”는 리스크가 크다.

#### 메타데이터도 아직 데모 톤이 강하다

- `apps/web/app/layout.tsx`
  - description이 `코드 제출 직후 설명 면접이 어떻게 달라지는지 보여주는 ... 인터랙티브 데모`
- 발표용 사이트라면 나쁘지 않다.
- 그러나 실제 도입 검토용 사이트라면 `데모`보다 `통제형 기술면접 평가 시스템`과 `기관 운영 가치`가 먼저 노출돼야 한다.

## 메시지-구현 정합성 판정

| 항목 | 현재 상태 | 판정 |
| --- | --- | --- |
| 코테→면접 브릿지 | 상세 화면에서 매우 잘 보임 | 일치 |
| 질문 통제 구조 | 상세 화면에서 명확함 | 일치 |
| 코드 근거 기반 분기 | variant + evidence 패널로 드러남 | 일치 |
| 기관용 평가 시스템 포지셔닝 | 홈에서 거의 보이지 않음 | 충돌 |
| 운영 신뢰성 신호 | fallback 노출이 강해 신뢰 약화 | 약함 |
| 장기 제품성/도입 가능성 | 문서에는 있으나 사이트 번역 부족 | 약함 |

## 교육기관 기준 최종 판단

### 지금 바로 말할 수 있는 것

- 부트캠프 강사에게 “이 시스템은 같은 문제라도 학생 코드 품질에 따라 다른 꼬리질문을 던지고, 그 이유까지 보여준다”라고 설명하면 납득시킬 재료는 있다.
- 즉, `제품 핵심 메커니즘`은 사이트 안에 이미 존재한다.

### 아직 말하기 어려운 것

- 사이트만 보고 “이건 우리 교육 운영에 넣을 만하다”라는 판단을 끌어내기에는 부족하다.
- 이유는 세 가지다.
  - 첫인상이 개인 연습 서비스 톤이다.
  - 기관 가치가 홈에서 바로 보이지 않는다.
  - 실제 기본 체험이 fallback 문구와 영문 리포트 때문에 임시 데모처럼 느껴질 수 있다.

## Go / No-Go 기준

이번 버전에 대한 외부 검증 판정 기준은 아래로 둔다.

### Go

- 인터뷰 참가자의 절반 이상이 홈 화면만 30초 보고 아래 설명을 자발적으로 한다.
  - `코딩 테스트 직후 설명 능력을 평가하는 시스템`
  - `강사/부트캠프가 반복 면접 운영에 쓸 수 있는 도구`
- 대표 문제 데모 후 절반 이상이 아래 가치를 자발적으로 언급한다.
  - 사람 면접 시간 절감
  - 학생 약점 데이터 축적
  - 질문 근거가 보여서 신뢰 가능

### No-Go

- 절반 이상이 아래처럼 이해하면 현재 포지셔닝은 실패다.
  - `그냥 AI 모의면접 앱 같다`
  - `취준생 개인용 연습 서비스 같다`
  - `데모는 되는데 실제 운영 도구로는 아직 못 믿겠다`

## 인터뷰에서 꼭 확인할 질문

아래 항목은 `docs/validation_interview_script.md`에서 그대로 사용한다.

1. 첫 화면만 보고 이 제품을 한 문장으로 설명해 달라.
2. 누가 돈을 내고 도입할 제품으로 보이는지 말해 달라.
3. 대표 문제 화면에서 가장 신뢰 가는 요소와 가장 불안한 요소를 말해 달라.
4. 이걸 커리큘럼 어디에 넣을지 말해 달라.
5. 지금 상태로 실제 강사/기관에게 보여줘도 되는지 `예 / 아니오`로 답해 달라.

## 즉시 수정 우선순위

### 우선순위 1

- 홈 첫 화면을 `개인 연습 서비스`에서 `기관용 평가 시스템` 톤으로 재정렬
- 바꿔야 할 핵심 요소
  - 브랜드/헤드라인
  - 보조 카피
  - 사이드 카드 2개
  - CTA 문구

### 우선순위 2

- fallback 경험을 외부 시연용 기준으로 정리
- 최소 조건
  - 영문 fallback summary 제거
  - 현재 모드가 fallback이어도 “왜 괜찮은지” 설명 가능한 카피로 변경
  - 모델 미연동 시에도 제품 품질이 임시처럼 보이지 않게 정리

### 우선순위 3

- 홈에서 기관 운영 가치를 직접 노출
- 추천 메시지
  - 강사 1명이 반복 기술면접에 쓰는 시간 절감
  - 수강생별 설명 약점 데이터 축적
  - 동일 문제, 다른 코드, 다른 질문 흐름

## 결론

현재 사이트는 `핵심 메커니즘 데모`로는 쓸 만하다.

하지만 `부트캠프 강사든 회사든 당장 보여줄 만한 사이트`라고 말하려면 아직 이르다.

- 보여줄 수는 있다.
- 다만 현재 버전은 `구조가 좋다`는 평가를 받기 쉽고, `바로 도입 검토할 만하다`는 평가까지는 못 간다.
- 다음 단계는 기능 추가보다 `포지셔닝 정렬`과 `신뢰 신호 보강`이 우선이다.
