# 현재 사이트 검증 메모

## 목적

- 현재 웹사이트가 `coding test to interview bridge`를 설득력 있게 보여주는지 검증한다.
- 특히 `교육기관/부트캠프` 관점에서 이 사이트가 실제로 도입 검토 가능한 수준의 제품처럼 보이는지 판단한다.
- 결과는 `지금 강한 점`, `신뢰를 깎는 지점`, `즉시 보완 우선순위`로 정리한다.

## 검증 기준

이번 검증은 아래 5개 축으로 고정한다.

| 축 | 질문 | 점수 |
| --- | --- | --- |
| 문제 인식 명확성 | 첫 화면만 보고 무엇을 해결하는지 이해되는가 | 4/5 |
| 코테→면접 브릿지 전달력 | 코드 제출 직후 설명 면접으로 이어지는 흐름이 자연스러운가 | 4/5 |
| 기관 구매자 가치 | 강사/부트캠프 운영자가 왜 필요한지 바로 보이는가 | 4/5 |
| 운영 신뢰성 | 질문 생성, 평가 방식, fallback 처리까지 믿을 만해 보이는가 | 3/5 |
| 데모 완결성 | 대표 시나리오가 중간 붕괴 없이 끝까지 완주되는가 | 3/5 |

## 한 줄 판정

현재 사이트는 `핵심 데모는 성립하고 메시지도 많이 정리됐지만, QA 신뢰성과 운영형 근거는 더 보강해야 한다`.

- 강한 부분은 대표 문제 상세 화면이다.
  - `코드 제출 → 질문 분기 → 질문 근거 → 3축 피드백`이 한 화면에서 이어진다.
  - 같은 문제에 다른 코드를 넣었을 때 질문 흐름이 달라진다는 데모 포인트가 명확하다.
- 약한 부분은 홈/랜딩이다.
  - 예전보다 훨씬 기관용 톤에 가까워졌지만, 여전히 운영 근거를 더 구조적으로 보여줄 여지가 있다.
  - 특히 “도입 가능한 시스템”으로서의 신뢰는 UI보다 검증 체계와 운영 데이터 구조에서 더 보강돼야 한다.

## 근거 요약

### 1. 강한 지점

#### 문제 상세 화면의 제품 핵심은 잘 보인다

- `apps/web/components/problem-workspace.tsx`
  - `FLOW_META`로 `일반 심화 질문 / 확장 검증 질문 / 개념 복구 질문`을 명시한다.
  - `질문 근거`, `코드 분석 요약`, `왜 이런 질문이 나왔는지` 섹션이 있어 질문 제어 구조가 드러난다.
  - 대표 문제에서는 variant 버튼으로 `느린 풀이 / 안정적인 풀이 / 오류 코드`를 비교하게 되어 있어 데모 메시지가 선명하다.
- 이 구조는 문서에서 강조한 차별점과 정합적이다.
  - `docs/구현_제안서.md`: `통제형 질문 생성`, `코드 근거 기반 분기`, `3축 평가`
  - `docs/구현_수정안.md`: `AI 면접 앱처럼 보이는 요소보다 코드 근거 기반 분기와 질문 통제를 먼저 보이게 해야 한다`

#### 대표 사용자 흐름은 실제로 끝까지 돈다

- 실행 근거: 화면 구조, 단위 테스트, `npm run verify`
- 확인된 사실
  - 홈에서 대표 CTA가 노출된다.
  - 대표 문제에서 코드 제출 후 `일반 심화 질문`이 시작된다.
  - 4턴 답변 뒤 `3축 피드백`까지 완료된다.
- 해석
  - 현재 구현은 “보여주려는 이야기”를 코드와 단위 테스트 기준으로는 끝까지 증명할 수 있는 수준이다.
  - 다만 브라우저 E2E는 별도로 계속 성공해야 하므로, 완결성 평가는 QA 파이프라인 상태와 함께 봐야 한다.

### 2. 약한 지점

#### 홈이 기관용 시스템이 아니라 개인 연습 서비스처럼 보인다

- `apps/web/app/page.tsx`
  - 브랜드가 `Socratic Deep-Dive`로 정리돼 있다.
  - 핵심 카피와 proof strip이 `세션 비용`, `강사 시간 절감`, `질문 통제`를 직접 보여준다.
  - `기관 운영 비교`와 `질문 통제 구조`가 홈에서 바로 노출된다.
- 문제는 방향성이 틀렸다는 것이 아니라 우선순위가 틀렸다는 점이다.
  - 현재 문서의 1차 고객은 `교육기관/부트캠프`다.
  - 지금은 개인 학습 도구처럼 읽히는 문제는 많이 줄었고, 남은 과제는 `기관 운영 근거를 얼마나 믿게 하느냐`에 더 가깝다.

#### 기관 구매자 가치가 랜딩에서 거의 보이지 않는다

- 현재 홈은 아래 메시지까지는 이미 보여준다.
  - 왜 강사가 사람 면접관 시간을 줄일 수 있는가
  - 왜 질문 정확도를 통제 구조로 설명할 수 있는가
- 다만 아래 메시지는 아직 약하다.
  - 왜 cohort 단위 운영 자산이 되는가
  - 왜 운영 시스템으로 장기간 쓸 수 있는가
- 문서에는 이미 이 메시지가 정리돼 있다.
  - `docs/구현_제안서.md`: 기관 운영 가능성, 약점 데이터 축적, 운영 비용 관리
  - `docs/구현_수정안.md`: 기관 운영 pain을 먼저 보여야 한다
- 즉, 제품 정의의 1차 번역은 됐고, 운영 스토리의 2차 번역이 아직 부족한 상태다.

#### 현재 기본 데모 경험은 fallback 의존성이 크고, 그 사실이 신뢰를 깎는다

- `apps/api/app/services/llm_service.py`
  - 모델 호출 실패 시 질문은 deterministic fallback으로, 리포트는 heuristic fallback으로 내려간다.
  - fallback summary는 현재 한국어 안전 모드 문구로 정리돼 있다.
- 해석
  - 내부 QA 관점에서는 안전장치가 맞다.
  - 하지만 외부 시연이나 기관 검토 기준에서는 “기본 체험이 deterministic 모드일 때도 충분히 제품답게 보이는가”를 계속 관리해야 한다.

#### 현재 가장 현실적인 리스크는 기능 부재보다 QA 파이프라인 신뢰성이다

- `docs/qa_plan.md`
  - 브라우저 E2E는 프로덕션 빌드 후 Playwright 기준으로 검증하도록 설계돼 있다.
- 2026-04-23 기준 점검 결과
  - `npm run verify`는 통과했다.
  - `npm run qa:e2e`는 복구 작업을 거쳐 다시 기준선으로 사용해야 한다.
- 해석
  - 현재 신뢰를 가장 크게 깎는 것은 “기능이 없다”가 아니라 “브라우저 기준 완결성 검증이 항상 보장되느냐”다.

#### 메타데이터도 아직 데모 톤이 강하다

- `apps/web/app/layout.tsx`
  - description이 `통제형 기술면접 평가 시스템` 중심으로 정리돼 있다.
- 메타데이터 자체의 방향은 현재 홈 메시지와 정합적이다.
- 남은 과제는 메타데이터보다 실제 운영 신뢰 신호를 더 쌓는 것이다.

## 메시지-구현 정합성 판정

| 항목 | 현재 상태 | 판정 |
| --- | --- | --- |
| 코테→면접 브릿지 | 상세 화면에서 매우 잘 보임 | 일치 |
| 질문 통제 구조 | 상세 화면에서 명확함 | 일치 |
| 코드 근거 기반 분기 | variant + evidence 패널로 드러남 | 일치 |
| 기관용 평가 시스템 포지셔닝 | 홈에서 직접 드러남 | 대체로 일치 |
| 운영 신뢰성 신호 | deterministic 안전장치는 보이지만 QA 신뢰성 보강 필요 | 약함 |
| 장기 제품성/도입 가능성 | 문서에는 있으나 운영 스토리 번역은 아직 부족 | 약함 |

## 교육기관 기준 최종 판단

### 지금 바로 말할 수 있는 것

- 부트캠프 강사에게 “이 시스템은 같은 문제라도 학생 코드 품질에 따라 다른 꼬리질문을 던지고, 그 이유까지 보여준다”라고 설명하면 납득시킬 재료는 있다.
- 즉, `제품 핵심 메커니즘`은 사이트 안에 이미 존재한다.

### 아직 말하기 어려운 것

- 사이트만 보고 “이건 우리 교육 운영에 넣을 만하다”라는 판단을 끌어내기에는 부족하다.
- 이유는 세 가지다.
  - 운영 데이터가 어디에 축적되고 어떻게 활용되는지까지는 아직 보이지 않는다.
  - 세션 저장과 운영 기능이 아직 MVP 코어 바깥에 있다.
  - 브라우저 E2E 검증 체계가 늘 같은 품질로 보장된다는 신호가 더 필요하다.

## Go / No-Go 기준

이번 버전에 대한 외부 검증 판정 기준은 아래로 둔다.

### Go

- 인터뷰 참가자의 절반 이상이 홈 화면만 30초 보고 아래 설명을 자발적으로 한다.
  - `코딩 테스트 직후 설명 능력을 평가하는 시스템`
  - `강사/부트캠프가 반복 면접 운영에 쓸 수 있는 도구`
- 대표 문제 데모 후 절반 이상이 아래 가치를 자발적으로 언급한다.
  - 사람 면접 시간 절감
  - 학생 약점 데이터 축적
  - 질문 근거가 보여서 신뢰 가능

### No-Go

- 절반 이상이 아래처럼 이해하면 현재 포지셔닝은 실패다.
  - `그냥 AI 모의면접 앱 같다`
  - `데모는 되는데 실제 운영 도구로는 아직 못 믿겠다`

## 인터뷰에서 꼭 확인할 질문

아래 항목은 `docs/validation_interview_script.md`에서 그대로 사용한다.

1. 첫 화면만 보고 이 제품을 한 문장으로 설명해 달라.
2. 누가 돈을 내고 도입할 제품으로 보이는지 말해 달라.
3. 대표 문제 화면에서 가장 신뢰 가는 요소와 가장 불안한 요소를 말해 달라.
4. 이걸 커리큘럼 어디에 넣을지 말해 달라.
5. 지금 상태로 실제 강사/기관에게 보여줘도 되는지 `예 / 아니오`로 답해 달라.

## 즉시 수정 우선순위

### 우선순위 1

- QA 파이프라인을 안정적으로 유지
- 바꿔야 할 핵심 요소
  - 프로덕션 빌드 기준 E2E 재현성
  - Playwright 서버 설정
  - 문서와 실제 검증 결과의 일치

### 우선순위 2

- 운영형 근거를 더 직접 노출
- 최소 조건
  - 세션 데이터 축적 방향
  - 향후 운영 대시보드/강사 활용 연결
  - 현재가 코어 데모 범위라는 점의 명확한 설명

### 우선순위 3

- deterministic 안전 모드의 신뢰 설명 강화
- 추천 메시지
  - 왜 fallback이어도 질문 통제 구조 증명에는 충분한가
  - 어떤 조건에서 LLM 모드로 자연스럽게 확장되는가
  - 무엇이 현재 구현이고 무엇이 Post-MVP인지

## 결론

현재 사이트는 `핵심 메커니즘 데모`로는 충분히 쓸 만하다.

하지만 `부트캠프 강사든 회사든 당장 보여줄 만한 사이트`라고 말하려면 아직 이르다.

- 보여줄 수는 있다.
- 다만 현재 버전은 `질문 통제 구조는 좋다`는 평가를 받기 쉽고, `운영형 제품으로 바로 도입 검토하겠다`까지 가려면 추가 근거가 더 필요하다.
- 다음 단계는 기능 추가보다 `QA 신뢰성 복구`, `운영 신호 보강`, `현재 구현 범위와 미래 범위의 명확한 분리`가 우선이다.
