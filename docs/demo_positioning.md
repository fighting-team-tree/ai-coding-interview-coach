# 데모 포지셔닝 메모

## 목적

- 현재 웹 데모가 `실서비스 전체 화면`이 아니라 `코어 워크스페이스를 압축한 시연 화면`이라는 점을 명확히 정리한다.
- 발표, 데모 영상, 질의응답에서 반복적으로 사용할 표준 설명을 고정한다.
- `실제 서비스는 어떻게 할 건데?`, `서비스 화면하고 너무 다른 것 아닌가?` 같은 질문에 일관되게 답할 수 있도록 한다.

## 핵심 판단

이번 대회 단계에서 먼저 증명해야 하는 것은 `서비스 외형`이 아니라 아래 3가지다.

1. 같은 문제라도 코드가 다르면 질문이 실제로 달라지는가
2. 그 질문이 코드와 문제 근거 안에서 통제되는가
3. 면접 종료 후 결과가 3축 피드백 자산으로 남는가

따라서 현재 데모는 전체 서비스를 모두 펼쳐 보여주기보다, 가장 중요한 엔진이 작동하는 `코어 워크스페이스`를 압축해서 보여주는 방식이 맞다.

## 현재 데모 화면의 정의

현재 화면은 `실서비스 전체`가 아니라 아래 두 레이어를 합쳐 압축한 시연 화면이다.

- `문제 제출 이후의 인터뷰 워크스페이스`
- `질문 근거와 피드백 결과를 확인하는 평가 워크스페이스`

즉, 지금 보여주는 것은 서비스의 껍데기가 아니라, 제품 차별점이 실제로 발생하는 중심 구간이다.

권장 표현:

> 지금 보여드리는 화면은 전체 서비스 랜딩이 아니라, 실제 서비스 안에서 가장 중요한 코어 워크스페이스를 압축한 시연 화면입니다. 이번 대회에서는 예쁜 서비스 외형보다, 코드에 따라 질문과 피드백이 어떻게 달라지는지를 먼저 증명하는 데 집중했습니다.

## 왜 이런 형태로 보여주는가

이 제품의 차별점은 `AI 면접 앱처럼 대화하는 것`이 아니다.

- 질문이 문제별 Fact/Trap/Forbidden Boundary 안에서 통제된다는 점
- 제출 코드의 AST 신호에 따라 질문 흐름이 바뀐다는 점
- 질문과 피드백의 근거를 역추적할 수 있다는 점

이 세 가지가 핵심이다.

따라서 대회 시연에서는 아래 순서를 우선한다.

1. 동일 문제
2. 서로 다른 코드
3. 코드 프로파일 차이
4. 질문 플로우 차이
5. 3축 리포트 차이
6. 기관이 얻는 운영 자산 설명

즉, `한 사람이 AI와 대화하는 장면`보다 `같은 문제인데 코드가 다르면 질문이 다르게 나온다`를 보여주는 것이 우선이다.

## 실서비스와의 관계

실서비스는 현재 데모 화면 하나로 끝나지 않는다. 현재 시연 화면은 아래 구조 중 `2번`과 `3번`에 해당하는 코어 부분이다.

1. 홈/문제 탐색
2. 코딩 테스트 제출 또는 과제 제출
3. 코드 근거 기반 면접 워크스페이스
4. 세션 리포트 히스토리
5. 학습자 성장 추적
6. 기관용 cohort/강사 대시보드

권장 표현:

> 실제 서비스에서는 문제 탐색, 세션 히스토리, 학습자 성장 추적, 기관용 운영 대시보드가 이어집니다. 지금 시연하는 화면은 그중에서도 제품 차별점이 가장 강하게 드러나는 제출 이후 코어 워크스페이스입니다.

## 질문 대응 가이드

### 질문 1

`실제 서비스 화면하고 너무 다른 것 아닌가요?`

권장 답변:

> 네, 현재 화면은 전체 서비스 UI를 모두 펼친 버전이 아니라 코어 평가 워크스페이스를 압축한 시연 버전입니다. 이번 단계에서는 서비스 외형보다, 코드에 따라 질문과 피드백이 실제로 달라지는 핵심 엔진을 먼저 검증하는 것이 우선이라고 판단했습니다.

### 질문 2

`그럼 실제 서비스는 어떻게 확장되나요?`

권장 답변:

> 실제 서비스에서는 이 엔진을 중심으로 문제 탐색, 세션 리포트 히스토리, 학습자 성장 추적, 기관용 cohort/강사 운영 화면이 붙는 구조입니다. 현재 데모는 그중 제품 차별점이 가장 강한 코어 화면만 먼저 증명하고 있습니다.

### 질문 3

`지금은 그냥 데모 같고 서비스 같지는 않은데요?`

권장 답변:

> 맞습니다. 의도적으로 데모형 구성을 택했습니다. 다만 목적은 단순 시각 데모가 아니라, 질문 통제 구조와 코드 기반 분기 로직이 실제로 작동한다는 점을 증명하는 것입니다. 대회 단계에서는 이 핵심이 먼저 설득되어야 이후 서비스 레이어 확장도 의미가 있습니다.

## 발표/영상용 핵심 문장

- `지금 화면은 전체 서비스가 아니라, 서비스의 핵심 엔진이 작동하는 코어 워크스페이스입니다.`
- `이번 대회에서는 서비스 껍데기보다 코드에 따라 질문이 달라지는 구조를 먼저 증명했습니다.`
- `실서비스에서는 이 엔진 위에 문제 탐색, 세션 히스토리, 학습자 추적, 기관 대시보드가 붙는 구조입니다.`
- `즉, 지금은 기능을 못 만든 것이 아니라, 무엇을 먼저 증명해야 하는지 우선순위를 선택한 상태입니다.`

## 마지막 슬라이드/영상 마감 문구

추천 마감 구조:

1. `현재 시연 범위: 코어 워크스페이스`
2. `이미 증명한 것: 코드 기반 질문 분기 / 질문 통제 / 3축 피드백`
3. `실서비스 확장: 문제 탐색 / 세션 히스토리 / 학습자 성장 추적 / 기관용 운영 대시보드`

추천 한 줄:

> 오늘 보여드린 것은 서비스 전체가 아니라, 서비스의 핵심 엔진입니다. 이 엔진 위에 학습자 경험과 기관 운영 레이어를 확장하는 것이 실제 서비스 로드맵입니다.

## 사용 원칙

- 현재 구현된 것과 로드맵을 섞어 말하지 않는다.
- 현재 시연 화면은 `코어 워크스페이스`, 아직 붙지 않은 화면은 `실서비스 확장 레이어`로 구분해서 말한다.
- `서비스처럼 안 보인다`는 지적을 방어하지 말고, `핵심 엔진을 먼저 증명하기 위해 의도적으로 압축했다`는 논리로 흡수한다.
