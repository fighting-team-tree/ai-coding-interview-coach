# GitHub Commit Convention 및 대화형 커밋 워크플로우

에이전트는 사용자를 대신하여 프로젝트 코드를 커밋(`git commit`)할 때, 항상 아래 워크플로우를 최우선으로 따라야 합니다. 이는 변경된 기능을 논리적으로 묶어 커밋 히스토리를 깔끔하게 관리하고, 최종 결과물에 대한 결정권을 사용자에게 보장하기 위함입니다.

## 1. 커밋 전 상태 확인 (Prerequisites)
- `git status`를 통해 staged / unstaged 상태의 파일 목록을 파악합니다.
- 필요한 경우 `git diff`를 통해 변경된 코드 구조와 의미를 파악합니다.

## 2. 논리적 단위 분할 및 GitHub 컨벤션 적용
변경된 파일들을 성격별로 분류하여 **Conventional Commits** 규칙에 맞게 그룹화합니다.
- `feat:` 새로운 기능 추가
- `fix:` 버그 수정
- `docs:` 문서 수정 (README, 기획 문서 등)
- `refactor:` 코드 리팩토링 (기능 변화 없이 구조만 개선)
- `style:` 코드 포맷팅, 세미콜론 누락 수정 등 (로직 변경 없음)
- `test:` 테스트 코드 추가 및 수정
- `chore:` 빌드 설정, 패키지 매니저 관리 등

## 3. 사용자에게 커밋 옵션 제안 (AskUserQuestion 방식)
자동으로 커밋을 실행하지 말고, 아래와 같이 최소 2~3가지의 커밋 옵션을 사용자에게 명확히 제시한 후 답변을 대기해야 합니다.

* **옵션 1: 논리적 단위 세분화 (추천)** - 컨벤션에 따라 각기 다른 커밋 메시지로 최대한 깔끔하게 나누어 커밋 (예: docs 개별 커밋 + feat 개별 커밋 + refactor 개별 커밋)
* **옵션 2: 큰 기능 단위로 이분화** - 연관성이 큰 기능과 문서 구조를 묶어서 1~2개의 커밋으로 처리
* **옵션 3: 단일 커밋 통일** - 하나의 거대한 세션 단위로 한 번에 통합 커밋

**에이전트 행동 지침:**
> "커밋을 진행하기 위해 다음과 같이 커밋 단위를 분석했습니다. 어떤 옵션으로 커밋을 진행할까요? (1, 2, 3 번호로 선택해 주시거나 다른 의견을 남겨주세요.)"

## 4. 사용자 선택에 따른 실행
사용자가 선택한 옵션(혹은 직접 제안한 단위)에 맞춰 `git add <경로>`와 `git commit -m "메시지"`를 실행하여 버전을 기록합니다.
