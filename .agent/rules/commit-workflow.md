# GitHub Commit Convention 및 승인 기반 커밋 워크플로우

에이전트는 사용자를 대신하여 프로젝트 코드를 커밋(`git commit`)할 때, **절대 임의로 커밋을 실행하지 말고**, 항상 아래 워크플로우에 따라 플래닝 모드의 Implementation Plan(구현 계획)을 활용해 사용자의 사전 검토 및 승인을 받아야 합니다.

## 1. 커밋 전 상태 파악
- `git status`와 `git diff`를 통해 변경된 파일 목록과 상세 내용을 분석합니다.

## 2. 논리적 단위 분할 
변경 사항을 파악한 후, **Conventional Commits** (feat, fix, docs, refactor, style, test, chore 등) 규칙을 적용하여 파일들을 논리적인 커밋 단위로 그룹화하고, 가장 권장하는 형태의 분할(작업 내역) 계획을 세웁니다.

## 3. 커밋 계획 브리핑 및 승인 요청 (가장 중요)
채팅으로 단순 커밋 여부를 묻는 것에 더해, **Implementation Plan** (또는 작업 계획 요약) 형태로 이번 커밋 분할 방식과 각 커밋 메시지를 사용자에게 상세히 브리핑합니다.
- 브리핑 내용: 이번 커밋에 포함할 커밋 명칭, 관련된 주요 파일들, 각 커밋의 목적 명시.
- 이 계획이 타당한지 사용자에게 승인(명시적인 동의나 `진행해줘` 등의 답변)을 요청합니다. 버튼이 노출되는 환경이라면 버튼 UI를 적극 활용합니다.

## 4. 사용자 최종 승인 후 기계적 실행
사용자가 브리핑된 계획을 검토하고 승인을 밝히면, 그 즉시 합의된 파일 단위로 `git add`와 `git commit`을 순차적으로 자동 실행합니다. (사용자가 분할 방식을 다르게 변경 요청하면 그에 맞춰 다시 묶어 진행합니다.)
