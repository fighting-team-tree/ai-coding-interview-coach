# 에이전트 상호작용 정책 (Agent Interaction Policy)

이 규칙은 에이전트가 복잡한 작업을 수행할 때 사용자 경험과 작업 정확성을 보장하기 위해 반드시 지켜야 할 "생각과 행동의 순서"를 규정합니다.

---

## 1. 기본 원칙 (Thinking Before Doing)
에이전트는 사용자의 요청을 수신했을 때, 그 작업이 시스템에 미치는 영향이 크거나(커밋, 파일 다수 수정, 설정 변경 등) 명확한 설계가 필요한 경우 즉시 실행(Action)에 옮기지 않습니다.

## 2. 작업 프로세스 (Standard Workflow)
모든 주요 작업은 반드시 다음 4단계를 거칩니다.

1.  **현황 및 의도 파악 (Research/Status)**: 현재 시스템의 상태(git status, 파일 내용 등)를 확인하고 사용자의 의도를 정확히 이해합니다.
2.  **기획 및 브리핑 (Planning/Briefing)**: `Implementation Plan` 아티팩트를 사용하여 **구체적인 작업 단계, 수정할 파일, 예상 결과**를 사용자에게 브리핑합니다.
3.  **최종 승인 대기 (Wait for Approval)**: 사용자에게 "이 계획대로 진행해도 될까요?"라고 묻고, 사용자의 명시적인 동의(승인)가 있을 때까지 대기합니다.
4.  **기계적 실행 및 보고 (Execution/Follow-up)**: 승인이 완료된 후 기계적으로 작업을 수행하고, `Walkthrough`를 통해 결과를 보고합니다.

## 3. "진행해줘" 지침 (The "Proceed" Rule)
사용자가 최초 요청 시 "커밋 진행해줘", "바로 수정해줘" 등의 **미리 승인된 워딩**을 사용했더라도, 에이전트는 **2단계(브리핑)와 3단계(승인 대기)를 절대 생략할 수 없습니다.**

- **이유**: 사용자가 생각하는 '커밋 단위'나 '수정 범위'가 에이전트의 판단과 다를 수 있기 때문입니다.
- **행동 요령**: "요청하신 대로 진행하겠습니다. 다만, 커밋/수정 전에 제가 가진 계획을 먼저 보여드리고 승인받은 후 진행하는 것이 프로젝트 규칙입니다. 아래 계획을 확인해 주세요."라고 안내합니다.

## 4. 동시 실행 금지 (No Joint Operations)
기획 단계(Implementation Plan 생성)와 실행 단계(Tool Call을 통한 실제 수정/커밋)를 한 번의 턴(Turn)에 동시에 수행하지 않습니다. 반드시 사용자의 동의 확인을 위한 **턴의 분리(Turn Isolation)**가 이루어져야 합니다.

---

💡 **핵심 가치**: "속도보다 정확성, 독단보다 합의"를 최우선으로 합니다.
